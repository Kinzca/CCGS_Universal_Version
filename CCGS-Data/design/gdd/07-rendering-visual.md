# Game Design Document: 视觉表现与认知负荷

> **系统：** Rendering, Visual Design & Cognitive Load Control  
> **版本：** 1.0  
> **设计：** Game Designer

## 1. 视觉风格 (Visual Style)

### 1.1 风格定位

| 维度 | 设计决策 |
| :--- | :--- |
| 整体基调 | 简洁几何风格，强调空间关系的**可读性** |
| 色彩策略 | 功能性配色优先于装饰性 |
| 材质质感 | 清晰的块面区分，避免过度纹理干扰空间判断 |
| 光影 | 利用光影强化空间深度感知 |

### 1.2 路径引导视觉

| 引导元素 | 用途 |
| :--- | :--- |
| **绿色泛光提示** | 标识关键路径方块，引导视线 |
| 深度光影 | 利用空间深度差异直观引导玩家 |
| 空位视觉 | 空位 slot 不渲染 block，形成明确的"缺口"感 |

## 2. 认知负荷控制 (Cognitive Load Theory → Design)

### 2.1 三种认知负荷

根据 Sweller 认知负荷理论（CLT），3D 解谜中的认知负荷分为：

| 类型 | 定义 | 在游戏中的表现 | 设计策略 |
| :--- | :--- | :--- | :--- |
| **内在负荷** | 任务本身的难度 | 旋转步数、多轴复合度 | 由 DDA 的 M 值控制 |
| **外在负荷** | 糟糕设计带来的多余负担 | UI杂乱、摄像机不稳、视觉遮挡 | **必须最小化** |
| **相关负荷** | 促使玩家建立心智模型的"有益挣扎" | 理解旋转规律、路径推演 | 通过引导**最大化** |

### 2.2 降低外在负荷

| 策略 | 实现 |
| :--- | :--- |
| UI简洁 | 只保留必要信息，避免信息过载 |
| 减少视觉干扰 | 动态减少非关键方块的视觉权重 |
| 摄像机控制平滑 | 避免"摄像机摩擦"消耗认知资源 |
| 旋转动画清晰 | 旋转过程中保持空间关系可追踪 |

> **"摄像机摩擦" (Camera Friction)**：若摄像机控制不佳，玩家的认知资源会从空间推理转移到镜头控制上，导致迅速受挫。

### 2.3 最大化相关负荷

| 策略 | 实现 |
| :--- | :--- |
| 泛光提示 | 绿色发光引导关键路径 |
| 路径高亮 | DDA降级时增加主轴引导 |
| 视觉一致性 | 相同类型方块保持统一视觉语言 |
| 深度线索 | 利用3D空间的纵深感强化空间理解 |

## 3. 方块渲染系统

### 3.1 方块视图工厂

`CubeViewFactorySys` 负责方块视图创建与管理：

| 职责 | 说明 |
| :--- | :--- |
| 视图创建 | 根据 `BlockModelInfo` 实例化视图 |
| 空位跳过 | `BlockModelInfo=null` 时不渲染 |
| 楼梯表现 | 独立的表现旋转修复逻辑 |
| 旋转动画 | 面旋转时的方块动画表现 |

### 3.2 渲染约束

| 约束 | 说明 |
| :--- | :--- |
| 空位不渲染 | 空位 slot 的 `BlockModelInfo=null` 时跳过视图创建 |
| 保留 slot 宇宙 | 完整 3×3×3 slot 保留，不破坏旋转/CoordConverter假设 |
| 楼梯朝向 | 旋转后楼梯的视觉朝向必须与逻辑朝向一致 |

### 3.3 窗口过渡动画 (Window Transition Animation)

为了强化无尽攀升的**强反馈原则**与空间感，在 `FormalWindowAdvance` 期间引入纯表现层的过渡动画系统（`TowerTransitionAnimator`）：

| 动画阶段 | 表现形式 | 认知意义 |
| :--- | :--- | :--- |
| **退场 (Drop-Out)** | 旧模型废墟（非底层）向下掉落至视图外后销毁 | 塔身逐渐崩解，强化"已经过去"的潜意识 |
| **入场 (Drop-In)** | 新增方块（中层、顶层）从视图外上方竖直掉落就位 | 错落有致的从下到上逐层搭建，暗示"攀升" |
| **原位保持** | 玩家角色与继承自旧顶层的新底层保持静止 | 为玩家提供空间安全感与视觉参照物锚点 |

> **零耦合设计**：动画器独立于核心逻辑系统，仅在切窗完成且新视图实例化后，接管特定 Transform 执行 `DOTween` 缓动，并在结束后释放操作锁。

### 3.4 操作失败反馈 (Action Fail Feedback)

为强化**强反馈原则**（"操作有回声"）和**可预期性**（"失败了但我知道为什么"），所有被拒绝的操作都通过 `ActionFailFeedbackSys` 提供即时视觉反馈：

| 失败类型 | 视觉表现 | 认知意义 |
| :--- | :--- | :--- |
| **旋转被拒绝** | 受影响面的方块沿旋转轴微旋转 6° → 弹性弹回原位（~0.25s）| "这个面被锁住了"——模拟齿轮卡住的物理反馈 |
| **移动被拒绝** | 玩家角色向意图方向前倾 0.12 单位 → 弹回原位（~0.25s）| "这个方向走不了"——人类通用的"试探碰壁"身体语言 |

> **设计区分**：两种失败反馈的**动画主体不同**（旋转=方块面 / 移动=角色），帮助玩家直觉区分失败类型。两者均不阻塞后续输入。

## 4. 空间认知与难度来源

### 4.1 难度因子分层

| 层级 | 难度因子 | 认知类型 | 心理旋转负荷 |
| :--- | :--- | :--- | :--- |
| L1 | 单轴旋转 | 空间感知 | 低 |
| L2 | 多轴复合旋转 | 空间推理 | 高 |
| L3 | 含视觉遮挡 | 空间想象 | 极高 |
| L4 | 多步序列组合 | 状态空间搜索 | 指数增长 |

### 4.2 DDA 降级时的视觉辅助

当 DDA 检测到玩家进入焦虑状态时，可触发的视觉辅助：

| 辅助手段 | 效果 | 理论依据 |
| :--- | :--- | :--- |
| 增加主轴引导 | 高亮建议旋转的面 | 降低多轴心理旋转难度 |
| 减少视觉干扰方块 | 淡化非关键块 | 降低外在认知负荷 |
| 路径泛光增强 | 更明显的路径提示 | 增强相关认知负荷 |

## 4.3 色彩主题系统 (Tower Color Theme)

### 设计理念

> **"色彩即心流的镜像"** — 环境色彩补偿玩家的情绪状态，而非强化它。

基于 **CLT 认知负荷理论** 的补偿式设计：内在负荷高时降低外在视觉刺激，内在负荷低时增加视觉丰富度。

### 双层配色模型

| 层级 | 驱动因子 | 控制维度 | 作用 |
| :--- | :--- | :--- | :--- |
| **第一层（主）** | DDA 节奏阶段 | 饱和度、明度、色温 | 决定色彩"感觉" |
| **第二层（辅）** | windowIndex 色相漫游 | 色相 (Hue) | 提供高度标识与彩虹回看 |

### 补偿式配色映射

| 节奏阶段 | 玩家状态 | 饱和度 | 明度 | 色温 | 视觉效果 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **BuildUp** | 正常探索 | **0.40** 中 | 0.60 标准 | 中性 | 温和自然 |
| **Peak** | 高压焦虑 | **0.30** 低 | **0.68** 亮 | 冷偏(-15°) | 淡雅可读 → 安抚 |
| **Relax** | 放松无聊 | **0.50** 高 | 0.55 深 | 暖偏(+15°) | 鲜艳饱满 → 刺激 |

> **饱和度底线 0.30**：文献 B1 确认低于此值面间色差消失，方块与背景融为一体。
> **Peak 明度 0.68**：文献 A+4 (Farnè 效应) 指出需与天空盒 (lit=0.88) 拉开 ΔL≥0.16 的距离。

### 面着色系统（等距三面方向光模拟）

基于文献 A★1（等距三面着色系统）模拟从右上方 45° 打入的方向光，并结合 A+1/A+16（Hue Shifting）亮面偏暖/暗面偏冷防止灰泥感：

| 面 | 色相偏移 | 明度偏移 | 设计意图 |
| :--- | :--- | :--- | :--- |
| **Top** | +0° | **+0.12** | 最亮面 — 直射光 |
| **Bottom** | +0° | **-0.20** | 最暗面 — 完全遮蔽 |
| **Front** | **+12°** (暖) | **+0.04** | 次受光面 — 45° 斜射 |
| **Back** | **-15°** (冷) | **-0.12** | 背光面 — 显著暗于 Front |
| **Right** | **+8°** (微暖) | **-0.02** | 侧光面 |
| **Left** | **-12°** (冷) | **-0.08** | 侧背光面 |

> **关键对比度**：Front(+0.04) vs Back(-0.12) = ΔL=0.16，满足 WCAG 1.4.11 非文本对比度 3:1 标准。

### 天空盒 Figure-Ground 分离

基于 A4 (Monument Valley) 和 A+4 (Farnè 效应)，确保方块从天空盒中自然"跳出"：

| 原则 | 实现 |
| :--- | :--- |
| 天空 sat < 方块 sat | skyTop = finalSat × 0.35（更淡） |
| 天空 lit > 方块 Top lit | skyTop lit = 0.88（更亮） |
| 方块为 figure，天空为 ground | 高对比度的 figure 从低饱和 ground 中凸显 |

### 品牌色

交错空间蓝 `HSL(200°, 0.40, 0.62) ≈ #7BB8D4`，作为 `HueStart = 200°`。每窗口色相步进 20°，约 18 窗口完成一次色环循环。

### 技术实现

- **方块**：`MaterialPropertyBlock` per-Renderer，每个窗口独立颜色，支持彩虹回看
- **天空盒**：`SharedMaterial` 全局唯一，Lerp 过渡 1.5s（EaseInOutQuad）
- **Colr 插件**：利用 `MasterShader` 6 面梯度着色 + `GradientSkybox` 双色渐变
- **对比度验证**：Editor 工具 `Tools → Tower Theme → Validate Contrast`，基于 WCAG 2.1 相对亮度公式自动验证

### 远景规划：塔下场景叙事

> 玩家在主界面通过滑动摄像机回看已攀爬的塔身。塔身下方点缀微缩场景装饰（花园、废墟、水池等），使用 Colr 风格化材质，颜色跟随该高度段的配色方案，形成"攀升旅途的回忆走廊"。

## 5. 调试可视化

### 5.1 对比度验证

Editor 工具 `TowerThemeContrastValidator`（路径：`Tools → Tower Theme → Validate Contrast`）基于 WCAG 2.1 标准验证所有节奏阶段下面间对比度。输出 WCAG 对比度比率 (CR)、相对亮度差 (ΔL) 和灰度值。

### 5.2 诊断日志

所有关键行为必须具备日志输出：
- 伤害计算（如适用）
- 状态切换
- 生成失败原因
- 路径来源标记
- `LegacyFallbackUsed` 兼容路径使用记录
